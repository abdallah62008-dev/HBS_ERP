## 2. Server Requirements

### Runtime versions

| Component | Minimum | Recommended | Source |
|---|---|---|---|
| PHP | 8.2 | **8.3.x** (matches local dev: 8.3.30) | `composer.json` `"php": "^8.2"` |
| Composer | 2.5 | 2.7+ | matches local Laragon |
| Node.js | 18 | **20 LTS** | `package.json` Vite 6 / React 18 |
| npm | 9 | 10 | bundled with Node 20 |
| MySQL | 8.0 | **8.4 LTS** | matches local dev |
| Web server | Nginx 1.22 / Apache 2.4 | **Nginx + php-fpm** | standard Laravel pattern |
| OS | Any Linux x86_64 | **Ubuntu 22.04 LTS** | well-supported by Laravel/Inertia |

### Required PHP extensions

Mandatory (Laravel 11 + this project):

- `pdo_mysql`, `mbstring`, `openssl`, `tokenizer`, `xml`, `ctype`, `json`,
  `bcmath`, `fileinfo`, `curl`, `dom`, `filter`, `hash`, `pcre`,
  `session`, `xmlreader`, `xmlwriter`, `simplexml`
- `gd` — required by `barryvdh/laravel-dompdf` for label PDFs
- **`zip`** — required by `maatwebsite/excel` (the local dev needed `extension=zip` enabled in `php.ini` per the original handoff)
- `intl` — recommended for locale-aware formatting

Verify with:
```bash
php -m | grep -E '^(pdo_mysql|mbstring|openssl|gd|zip|intl|bcmath|curl)$'
```

### Web server expectations

- Document root MUST point to `public/` — never to the project root.
- HTTPS terminated at the load balancer or Nginx; HTTP redirects to HTTPS.
- `client_max_body_size` ≥ **20M** (covers Excel imports + label PDFs + pre-shipping photos).
- PHP-FPM `upload_max_filesize` and `post_max_size` ≥ **20M**.
- `memory_limit` ≥ **256M** (Excel imports can spike).

### Queue and scheduler

- Queue is configured `database` driver. A `php artisan queue:work --tries=3 --max-time=3600` worker should run under **systemd** or **supervisor** to keep it alive.
- Laravel scheduler (`php artisan schedule:run`) should run from cron every minute:
  ```cron
  * * * * * cd /var/www/hnawbas && php artisan schedule:run >> /dev/null 2>&1
  ```

### Storage permissions

The Laravel-standard requirement applies:

```bash
sudo chown -R www-data:www-data storage bootstrap/cache
sudo chmod -R 775 storage bootstrap/cache
```

Per-directory write access is needed for: `storage/app`, `storage/app/public`,
`storage/framework`, `storage/logs`, `bootstrap/cache`. The
`storage/app/public/{attachments,backups,imports,shipping-labels,pre-shipping-photos}`
subfolders are auto-created on first use; they all sit behind the
`public/storage` symlink (created by `php artisan storage:link`).

## 5. Deployment Command Checklist

Execute as the deploy user (`www-data` or a dedicated `deploy` user). All commands assume `/var/www/hnawbas` as the deploy path — adjust as needed.

```bash
# 0. Pre-flight
php -v                                # PHP 8.2+
composer --version                    # Composer 2.5+
node --version && npm --version       # Node 20+, npm 10+
mysql --version                       # 8.0+
php -m | grep -E '^(pdo_mysql|gd|zip|mbstring|openssl|bcmath|intl|curl)$'

# 1. Clone repo (or pull on subsequent deploys)
sudo mkdir -p /var/www/hnawbas && sudo chown deploy:deploy /var/www/hnawbas
git clone <REPO_URL> /var/www/hnawbas
cd /var/www/hnawbas
git checkout 5920fb3                  # or the staging branch / tag

# 2. PHP dependencies (production-mode, no dev tools, optimized autoloader)
composer install --no-dev --optimize-autoloader --no-interaction --prefer-dist

# 3. Node + frontend bundle (devDependencies needed only at build time)
npm ci
npm run build
# Optional: prune devDeps after build to shrink disk
# rm -rf node_modules

# 4. Environment file
cp .env.staging.example .env
nano .env                             # fill DB password, APP_URL, etc.

# 5. Application key
php artisan key:generate --force

# 6. Storage symlink
php artisan storage:link

# 7. Database (staging is intentionally fresh — see §6)
php artisan migrate --force --seed

# 8. Cache config / routes / views (production-style)
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan event:cache

# 9. File permissions
sudo chown -R www-data:www-data storage bootstrap/cache public/build
sudo chmod -R 775 storage bootstrap/cache

# 10. Queue worker (systemd unit recommended; one-off start shown)
php artisan queue:work --tries=3 --max-time=3600 &

# 11. Scheduler (cron line shown earlier; install once per host)

# 12. Smoke test (see §9)
```

For **subsequent deploys** (after the first one), use:

```bash
cd /var/www/hnawbas
php artisan down --retry=60                  # maintenance mode
git fetch origin && git checkout <new-sha>
composer install --no-dev --optimize-autoloader --no-interaction
npm ci && npm run build
php artisan migrate --force                  # NO --seed on subsequent deploys
php artisan config:cache && php artisan route:cache && php artisan view:cache && php artisan event:cache
sudo systemctl restart php8.3-fpm
sudo systemctl restart hnawbas-queue.service # if using systemd for the worker
php artisan up
```

## 6. Database & Seeder Plan

### Recommended approach for the FIRST staging deployment

1. **Use a fresh staging schema.** Do NOT import production data into staging at this stage.
2. Create the schema and user:
   ```sql
   CREATE DATABASE hbs_erp_staging CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
   CREATE USER 'hbs_staging'@'%' IDENTIFIED BY '<STRONG_PASSWORD>';
   GRANT ALL ON hbs_erp_staging.* TO 'hbs_staging'@'%';
   FLUSH PRIVILEGES;
   ```
3. Run **`php artisan migrate --force --seed`** once — applies all 55 migrations and runs the 10 baseline seeders (Permissions, Roles, Settings, ExpenseCategories, ReturnReasons, MarketerPriceGroups, FiscalYear, Warehouses, ShippingCompanies, AdminUser).
4. The seeded admin (`admin@hbs.local` / `Admin@2026`) is for first login only — **rotate immediately** (see §8).
5. UAT/demo data is intentionally NOT bundled. Operators add demo products/orders manually after first login, OR you can layer a separate `StagingDemoSeeder` later (out of scope for this deployment).

### Subsequent deploys

- Run **`php artisan migrate --force`** without `--seed` (avoid re-running idempotent-but-noisy seeders on every deploy).
- Always **back up the DB** before migrations:
  ```bash
  mysqldump --single-transaction --routines --triggers \
    -u hbs_staging -p hbs_erp_staging \
    | gzip > /var/backups/hbs_erp_staging_$(date +%F-%H%M).sql.gz
  ```

### NEVER do without explicit approval

- `migrate:fresh` (drops all tables)
- `db:wipe`
- Importing production data into staging
- Seeding from a one-off script that touches staging-only data

## 8. Security Checklist

Run through this list at the end of EVERY staging deploy:

- [ ] `.env` file is **not** committed (`git status` is clean; `git ls-files | grep -E '^\.env$'` returns nothing).
- [ ] `APP_DEBUG=false` in the live `.env`.
- [ ] `APP_ENV=staging` (never `local` or `production` on this host).
- [ ] `APP_KEY` set (32 base64 bytes from `key:generate`).
- [ ] DB user uses a strong (≥20 char, mixed-class) password; user privileges scoped to the staging schema only.
- [ ] HTTPS enforced; HTTP redirects to HTTPS at Nginx.
- [ ] `SESSION_SECURE_COOKIE=true` and `SESSION_ENCRYPT=true`.
- [ ] `storage/`, `bootstrap/cache/` are writable only by `www-data` (775, not 777).
- [ ] No public access to `storage/app/private` or any path other than `public/`.
- [ ] **Rotate the seeded admin** immediately:
  ```bash
  php artisan tinker --execute="\App\Models\User::where('email','admin@hbs.local')->update(['email'=>'<your-real-email>','password'=>bcrypt('<NEW_STRONG_PASSWORD>')]);"
  ```
- [ ] Login route protected (Breeze defaults: rate-limit, throttle).
- [ ] Public registration disabled (verified at `routes/auth.php` — no `register` route exposed; `RegistrationTest` asserts both `/register` GET and POST return 404).
- [ ] PHP `upload_max_filesize` and Nginx `client_max_body_size` agree at ≤ 20M.
- [ ] Audit logs enabled (default; `AuditLogService` redacts password/token/secret keys before persisting).
- [ ] `fiscal_year_lock` middleware applied to: orders.update, orders.change-status, orders.destroy, orders.request-price-edit, expenses.update, expenses.destroy, purchase-invoices.update (verified via `php artisan route:list -vv`).
- [ ] `allow_negative_stock` setting seeded `false` (verified by SettingsSeeder).
- [ ] Self-approval block active (`ApprovalService::approve` refuses if requester == approver — covered by N5 negative test in UAT).
- [ ] Marketer data isolation active (`Order::scopeForCurrentMarketer` + `role:marketer,super-admin` middleware on `/marketer/*` routes — covered by N4).
- [ ] Server firewall: only 22 (admin SSH), 80 (→443 redirect), 443 open externally; MySQL 3306 bound to localhost only.
- [ ] Backups dir (`storage/app/public/backups`) NOT served publicly without auth (today they ARE — flagged in §11; mitigate via Nginx `deny` rule on `/storage/backups/`).

## 10. Rollback Plan

### Before the deploy

- Record the **current commit hash** that's running on staging (write it down; you can also `git rev-parse HEAD > /var/log/hnawbas-prev-commit.txt` as part of the deploy script).
- Take a fresh DB backup (see §6).

### To roll back code

```bash
cd /var/www/hnawbas
php artisan down --retry=60
git fetch origin
git checkout <previous-commit-sha>
composer install --no-dev --optimize-autoloader --no-interaction
npm ci && npm run build
php artisan config:clear && php artisan route:clear && php artisan view:clear
php artisan config:cache && php artisan route:cache && php artisan view:cache
sudo systemctl restart php8.3-fpm
sudo systemctl restart hnawbas-queue.service
php artisan up
```

### To roll back the database

Only do this if a migration introduced data loss / corruption. Otherwise prefer a forward-fix.

```bash
# Restore from the backup taken before the deploy
gunzip < /var/backups/hbs_erp_staging_<timestamp>.sql.gz \
  | mysql -u hbs_staging -p hbs_erp_staging
```

`php artisan migrate:rollback --force` is acceptable for the most recent
batch IF the down() methods are well-tested — but on a fresh staging DB,
restoring from `mysqldump` is faster and safer.

### Post-rollback verification

1. `php artisan migrate:status` — confirm schema matches the rolled-back code.
2. Re-run §9 smoke test (steps 1-3 minimum).
3. `tail -f storage/logs/laravel-*.log` for 5 minutes; watch for new errors.

## 11. Non-Blocking Known Concerns

These were surfaced during UAT and the inventory-fix pass. None block the staging deploy; track them as follow-ups.

1. **`marketer_product_prices` per-marketer override is not consulted at order creation.** `OrderService::buildItemRows` reads only `$variant->marketer_trade_price ?? $product->marketer_trade_price`. If/when per-marketer pricing is meant to win, add a lookup to `marketer_product_prices.trade_price` keyed by `(marketer_id, product_id, variant_id)`. Spec confirmation needed first.
2. **Pre-shipping photo `attachment_type` is case-sensitive.** Operators must pass exactly `Pre Shipping Photo`. The UI should hardcode this as a hidden form value rather than rely on operator input.
3. **Forced status jumps may leave reservations.** If an order goes `Confirmed → Delivered` (skipping `Shipped`), the Reserve movement persists. Realistic flows always go through Shipped, so this only manifests on admin-forced jumps.
4. **`brick/math` deprecation warnings** on every `decimal:2` cast. Cosmetic — math is correct. Awaiting upstream patch.
5. **Backup files publicly downloadable if URL known.** `storage/app/public/backups/*.zip` are served via the public storage symlink. URLs are not enumerable from outside but should still be access-controlled. Suggested mitigation: add an Nginx `location ~ ^/storage/backups/ { deny all; }` block, OR promote backups to be served via a controller with `permission:backup.manage` (Phase 6 hardening item).
6. **Item-level edits on placed orders are append-only via approval flow.** This is by design (handoff §5 #11); flagged here so support staff know not to expect inline item edits.
7. **Repository was just created (one commit).** No history yet. Once a CI pipeline is wired up, that commit becomes the baseline reference for diffs.
8. **Deferred modules intentionally NOT included**: full Tickets module, real barcode library, RTL, advanced charts, additional importers (orders/purchases/shipping-updates/marketers), full approval-handler migration, item-level order editing, mysqldump-based backups (currently JSON-zip).

## 12. Final Recommendation

**Proceed with the first staging deployment** using the checklist in §5, the database plan in §6, and the smoke test in §9. The system is at a stable baseline (commit `5920fb3`), the test suite + build are green, and the UAT confirmed every operational gate works correctly end-to-end.

**Critical path before the host goes live to UAT users:**

1. SSH into the staging host; install runtimes (§2).
2. Clone repo, run §5 commands.
3. Rotate the seeded admin password (§8).
4. Configure HTTPS + `SESSION_SECURE_COOKIE=true` (§8).
5. Add Nginx rule denying `/storage/backups/` (§11 #5) — recommended but not strictly blocking.
6. Run the §9 smoke test.
7. Hand over staging URL + rotated admin credentials to the UAT team via a secure channel (NOT email/Slack DM in plaintext).

Estimated time-to-staging from a clean Ubuntu 22.04 host: **45-60 minutes** including HTTPS setup.

**Do not promote to production from staging until:**

- §11 follow-ups #1, #2, #5 are resolved (or explicitly waived by the product owner).
- A backup-and-rollback dry-run has been performed.
- The `mysqldump`-based production backup strategy replaces the current JSON-zip backup (handoff §9 long-term task).
